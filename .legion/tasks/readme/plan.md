# README 增补重启提示与飞书机器人指南

## 任务元信息

- 风险分级: Low
- 规模分级: 非 Epic
- 标签: 无
- 设计策略: design-lite（不单独产出 RFC）

## 目标

更新 `README.md`，补充 LegionMind 来源说明、插件配置后的重启提示，以及可直接执行的飞书机器人配置/授权/成员/发布指南，并同步产出验证、评审和 PR 描述文档。

## 问题定义

当前 `README.md` 只有基础安装和配置说明，缺少三个对首次接入非常关键的提示：

1. 仓库前言没有说明该插件基于 `github:thrimbda/legion-mind` 的 vibe coding 方式完成，缺少项目来源背景。
2. AGENT 自动配置指南没有强调插件写入配置后需要重启 OpenCode，容易导致用户误以为配置未生效。
3. README 没有集中说明飞书机器人如何创建、开启机器人能力、授予最小必要权限、把目标用户纳入可用范围、在群中使用，以及后续如何发布新版本。

## 验收标准

- README 前几行新增项目来源说明，并明确提到 `github:thrimbda/legion-mind`。
- AGENT 自动配置指南中明确写出：配置插件后需要重启 OpenCode 才会加载新插件。
- README 新增飞书机器人配置指南，覆盖机器人配置、权限授予、用户/群接入与版本发布步骤。
- 指南内容与仓库现有实现保持一致：应用通过 `appId`/`appSecret` 获取 `tenant_access_token`，随后调用飞书 IM 消息接口发送消息。
- 产出 `test-report.md`、`review-code.md`、`report-walkthrough.md`、`pr-body.md`；Low 风险任务可不产出 `review-security.md`。

## 假设

- 仓库文档语言以中文为主，本任务继续沿用中文说明。
- 飞书平台当前仍要求应用开启机器人能力并发布版本后，消息发送相关配置才会对租户生效。
- 本插件仅依赖发送消息能力，不需要在 README 中扩展到通讯录、事件订阅或额外数据权限的实现说明。

## 约束

- 保持 `plan.md` 为摘要级契约，不把它扩展成 mini-RFC。
- README 改动只覆盖用户明确要求的前言、自动配置提醒、飞书机器人指南，不顺手重写其它章节。
- 指南内容优先引用仓库现有脚本、配置字段和发布流程，避免引入与代码不一致的承诺。
- 任务过程产物统一写入 `.legion/tasks/readme/`，不新增根目录长期文档。

## 风险说明

- 风险低：本次改动仅涉及文档和任务产物，不修改运行时代码、配置结构或外部接口。
- 主要风险是飞书开放平台术语或后台路径描述不准确，因此指南会以“最小必要权限 + 当前仓库实际调用链路”为准，并在必要处标注用户需按飞书后台实际文案操作。

## design-lite

### 方案摘要

README 采用“前言补背景 + 自动配置补生效提醒 + 新增独立机器人配置章节”的最小扩展方式：

1. 在标题下方增加一行项目来源说明，满足仓库背景披露要求。
2. 在 AGENT 自动配置指南中加入重启提示，并把“验证配置”改为重启后再验证，避免步骤断层。
3. 在 README 配置章节前后新增一节“飞书机器人配置指南”，按“创建应用 -> 开启机器人能力 -> 申请权限 -> 配置可用范围/添加用户 -> 获取接收者 ID -> 发布版本 -> 本仓库接入”的顺序组织。
4. 发布流程说明与仓库现有 `package.json`、`.github/workflows/release.yml`、历史文档中的版本发布步骤保持一致。

### 为什么这样做

- 改动集中在 README，一次性满足用户提出的三项文档缺口。
- 以接入顺序组织机器人指南，用户可以从创建应用一路走到实际发消息，减少跳读成本。
- 复用仓库已有脚本与发布方式，能降低文档与实现漂移风险。

### 非目标

- 本次不修改 `docs/QUICK_START.md` 或其它长期文档。
- 本次不新增自动化脚本来查询 `user_id` / `chat_id`。
- 本次不改变插件的配置字段、消息格式或发布机制。
